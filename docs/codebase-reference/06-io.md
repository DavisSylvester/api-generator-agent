# src/io/ -- File I/O, Workspace, and Reporting

Manages all filesystem operations, workspace layout, and report generation.

---

## workspace.mts

**Exports:** `Workspace`

**What it does:** Manages the filesystem layout for a pipeline run:

```
.workspace/<runId>/
  tasks/<taskId>/
    code/
    code-linted/
    tests/
    integration/
    iterations/<n>/code/
    status.json, qa-results.json, qa-knowledge.md
  output/
  docs/hoppscotch-collection.json
  logs/run.log
  .docs/
    progress.md
    tasks/<taskId>/activity.md
  plan.json, report.md, qa-knowledge.md
```

Key methods: `init()`, `initForResume()`, `initTask()`, `initIteration()`, `loadCompletedTaskIds()`, `planCachePath()`, plus path getters for every artifact type.

**Why it exists:** Enforces consistent filesystem layout. Enables resume (read previous statuses), debugging (navigate to specific iterations), and clean separation between observability (`.docs/`) and deliverables (`output/`).

---

## activity-log.mts

**Exports:** `ActivityEventType`, `ActivityEventInput`, `ActivityLog`

**What it does:** Append-only markdown activity log per task. Events recorded as markdown table rows with Timestamp, Event, Summary, Details, Artifact columns.

Key design:
- **Append-only:** Never rewrites history; safe to tail while running.
- **Idempotent header:** Table header written only on first event via `wx` flag.
- **Serialized writes:** Promise chain prevents concurrent interleaving.

Event types: `task-start`, `iteration-start`, `codegen-start/end`, `eslint-start/end`, `qa-start/end`, `iteration-end`, `task-end`, `note`.

**Why it exists:** Fine-grained observability per task for post-hoc debugging.

---

## progress-reporter.mts

**Exports:** `TaskStatus`, `TaskStateSnapshot`, `ProgressReporterInit`, `ProgressReporter`

**What it does:** Maintains a live `progress.md` file atomically rewritten on every event. Contains:
- Run metadata (ID, start time, elapsed, provider, config)
- Metrics (pass/fail counts, first-try rate, codegen/QA time, LLM calls/cost)
- Task sections by status (in-progress, failed, completed, pending)
- Links to per-task activity logs

**Why it exists:** Real-time visibility into pipeline progress. The user can `cat` or `tail` a single file to see status.

---

## report-generator.mts

**Exports:** `ReportData`, `GeneratedFileEntry`, `generateReport()`

**What it does:** Generates comprehensive final markdown report:
- Header table (run ID, duration, PRD size, task counts)
- Task plan table
- Observability links
- Task results with errors
- Integration test results
- Cost summary (total, per-model, per-task breakdowns)
- Generated files list

**Why it exists:** Single post-run document summarizing what was generated, what succeeded/failed, and how much it cost.

## File Operations

### file-protocol.mts
**Exports:** Re-exports `writeJson`, `readJson`, `writeCode`, `readCode`, `readAllCodeFiles`
**Why:** Barrel file for all file I/O utilities.

### read-code.mts
**Exports:** `readCode()`
**What it does:** Reads file text via `Bun.file().text()`, returns `Result<string, Error>`.

### read-json.mts
**Exports:** `readJson<T>()`
**What it does:** Reads and parses JSON via `Bun.file().json()`, returns `Result<T, Error>`.

### write-code.mts
**Exports:** `writeCode()`
**What it does:** Writes text to file, auto-creates parent directories, returns `Result<void, Error>`.

### write-json.mts
**Exports:** `writeJson<T>()`
**What it does:** Serializes as pretty-printed JSON and writes, auto-creates directories, returns `Result<void, Error>`.

### read-all-code-files.mts
**Exports:** `readAllCodeFiles()`
**What it does:** Recursively walks directory, reads all `.mts`, `.ts`, `.json`, `.toml`, `.mjs` files. Returns `Result<Map<string, string>, Error>`.
**Why:** After codegen produces files, the pipeline reads them all back for QA/ESLint/assembly.
