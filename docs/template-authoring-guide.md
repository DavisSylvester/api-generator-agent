# Template Authoring Guide

## Overview

agent-one uses a template system to generate code for each layer of a feature. Templates are raw TypeScript files (.tmpl.mts) that export render functions returning strings. No external template engine is required.

This guide explains how to create new addon templates that extend agent-one's generation capabilities.

---

## Template Contract (ITemplate)

Every template (base or addon) must implement the ITemplate interface:

```typescript
import type { IFeatureSpec } from "../core/interfaces/i-feature-spec.mts";

interface ITemplate {
  name: string;
  type: TemplateType;       // "base" | "addon"
  description: string;

  /** Files this template will generate, given a feature spec */
  plan(feature: IFeatureSpec): IGeneratedFile[];

  /** Render all files for the given feature */
  render(feature: IFeatureSpec, context: IGenerationContext): IRenderedFile[];

  /** Validate rendered output before writing */
  validate(files: IRenderedFile[]): IValidationResult;
}
```

### Key interfaces:

```typescript
interface IGeneratedFile {
  path: string;           // Relative path in the generated project
  description: string;    // What this file contains
}

interface IRenderedFile {
  path: string;           // Relative path in the generated project
  content: string;        // Full file content
}

interface IGenerationContext {
  projectName: string;
  outputDir: string;
  allFeatures: IFeatureSpec[];
}

interface IValidationResult {
  valid: boolean;
  errors: string[];
}
```

## Creating a New Addon Template

### Step 1: Create the addon directory

```
templates/addons/my-addon/
  index.mts
  my-addon.tmpl.mts
```

### Step 2: Implement the render function

Each .tmpl.mts file exports a function that accepts a feature spec (or other context) and returns a string of generated code:

```typescript
// templates/addons/my-addon/my-addon.tmpl.mts

import type { IFeatureSpec } from "../../src/core/interfaces/i-feature-spec.mts";

export interface MyAddonContext {
  feature: IFeatureSpec;
  projectName: string;
}

export function renderMyAddon(context: MyAddonContext): string {
  const { feature, projectName } = context;

  return `// Auto-generated by agent-one
// Project: ${projectName}
// Feature: ${feature.name}

import type { Logger } from "winston";

export class ${feature.name}Addon {

  private readonly logger: Logger;

  constructor(logger: Logger) {
    this.logger = logger;
  }

  public async execute(): Promise<void> {
    this.logger.info("${feature.name} addon executed");
  }
}
`;
}
```

### Step 3: Create the barrel export

```typescript
// templates/addons/my-addon/index.mts

export { renderMyAddon } from "./my-addon.tmpl.mts";
export type { MyAddonContext } from "./my-addon.tmpl.mts";
```

### Step 4: Register in the template registry

Add your addon to the template registry in `src/generation/template-registry.mts`. The registry discovers templates and makes them available to the generation engine.

## Template Best Practices

### Code conventions

All generated code must follow these conventions:

- **TypeScript strict mode** with .mts extension
- **Double quotes** for strings
- **.mjs** import specifiers (not .mts in import paths)
- **Named exports only** (no default exports)
- **Explicit return types** on all functions
- **No `any` type** -- use explicit types or `unknown`
- **Trailing commas** in multiline expressions
- **Blank line** after class opening brace
- **Kebab-case** file names
- **PascalCase** class and interface names
- **camelCase** variable and function names

### Template structure

1. **Start with imports** -- group by external, then internal
2. **Export interfaces/types first** -- they're consumed by other layers
3. **Use template literals** with `${expression}` for interpolation
4. **Handle edge cases** -- empty arrays, optional fields, no relationships
5. **Include JSDoc comments** on exported functions and classes

### Testing templates

Every template should have corresponding unit tests:

```typescript
// tests/templates/my-addon.test.mts
import { describe, it, expect } from "bun:test";
import { renderMyAddon } from "../../templates/addons/my-addon/my-addon.tmpl.mts";

describe("renderMyAddon", () => {
  it("should render valid TypeScript", () => {
    const output = renderMyAddon({
      feature: {
        name: "WorkOrder",
        domain: "work-orders",
        description: "Work order management",
        entities: [],
        dependsOn: [],
      },
      projectName: "test-api",
    });

    expect(output).toContain("WorkOrderAddon");
    expect(output).toContain("test-api");
  });
});
```

## Template Types

### Base templates

Base templates generate the standard CRUD stack for every feature:

| Template | Output | Layer |
|----------|--------|-------|
| interface | `i-<entity>.mts` | Data contracts |
| schema | `<entity>.validation.mts` | Zod validation |
| repository | `<entity>-repository.mts` | Data access |
| service | `<entity>-service.mts` | Business logic |
| router | `<entity>-router.mts` | HTTP layer |
| test | `<entity>-service.test.mts` | Unit tests |
| swagger-detail | `<entity>-swagger.mts` | API docs |

### Addon templates

Addon templates generate additional infrastructure:

| Template | Purpose |
|----------|---------|
| azure-terraform | Azure infrastructure as code |
| aws-cdk | AWS CDK constructs |
| queue-consumer | Message queue consumers |
| external-api-client | External API client wrappers |
| teams-notification | Microsoft Teams webhooks |
| timer-job | Scheduled/cron job scaffolding |

## IFeatureSpec Reference

The feature spec is the primary input to templates:

```typescript
interface IFeatureSpec {
  name: string;          // e.g., "WorkOrder"
  domain: string;        // e.g., "work-orders"
  description: string;   // e.g., "Work order management"
  entities: IEntitySpec[];
  dependsOn: string[];   // Names of features this depends on
}

interface IEntitySpec {
  name: string;          // e.g., "WorkOrder"
  pluralName: string;    // e.g., "WorkOrders"
  fields: IFieldSpec[];
  relationships: IRelationship[];
  operations: readonly string[];  // e.g., ["create", "read", "update", "delete", "list"]
}

interface IFieldSpec {
  name: string;
  type: string;          // "string" | "number" | "boolean" | "datetime" | "email" | etc.
  required: boolean;
  description?: string;
  defaultValue?: string;
  validation?: string;
}

interface IRelationship {
  targetEntity: string;
  type: "one-to-one" | "one-to-many" | "many-to-many";
  foreignKey: string;
  required: boolean;
}
```
